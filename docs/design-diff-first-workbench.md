# 设计定稿:工作台「diff 一等公民」机制

状态:**已定稿,待实施**。来源:2026-07-20 codev 会议裁决(sessions/2026-07-20-工作台如何把-diff-变成一等公民…/judge-card.md)+ 用户 2026-07-21 终审拍板。用户拍板优先于仲裁推荐。

## 采纳的架构(来自裁决,Claude 方案)

- **提案 = Revision**:每个 AI 的改动提案存"改前/改后"内容快照(CAS blob,按内容寻址),任何历史提案可完整还原、可审计;
- **硬门(deterministicGate)**:只做零误报的确定性判定——未授权执行者 / 基线漂移(文件在提案后被改过)/ blob 缺失(数据损坏)/ 沙箱应用失败。四条全过才可定稿,拦了就是真有问题;
- **软门 = 处置清单(disposition sheet)**:多提案触碰同一文件时强制进人工清单;
- **沙箱诚实原则**:无显式断言的沙箱只能证明"没崩",UI 必须明示"未被测试覆盖";
- **应用审计日志**(applications.jsonl):每次定稿落地记录工作区快照摘要、采纳的 Revision、逐条处置。

## 用户终审的三项决策(2026-07-21)

### 1. AI 间分歧的处置:不拦截、不止于警告——升格为决策时刻 ★ 定位级原则

AI 意见不一致(真冲突,或不同路径解同一问题)是**本产品的核心时刻**,处置方式:

- 机器可确定的问题走硬门(照拦);
- AI 判断出的逻辑/契约冲突:**不阻断**,在冲突项旁提供「深入讨论」入口(用户触发,不自动消耗调用),产出结构化 pro/con 对比(各方案收益、代价、风险),用户在 informed 前提下拍板;
- 原"advisory 黄灯"仅作为讨论入口的轻量形态,不是终态。
- (裁决原文的升格条件保留作参考:50 个接口变更任务实验、误报 <10% 才有资格考虑硬门——但按本决策,方向已从"要不要拦"转为"如何支撑讨论",实验优先级下调。)

### 1.5 每 agent 两级权限(用户 2026-07-21 补充明确)

工作台内每个参与者有两个**彼此独立**的权限位,均由用户显式授予、UI 可见可切换:

- **提案权(propose)**:是否允许该 agent 的发言携带 diff 提案(Revision)。默认关——讨论阶段全员只动嘴,何时给谁开由用户定(即红线 A 的"写开关")。
- **落地权(apply/finalize)**:是否允许该 agent 执行定稿动作(融合多 Revision、在副本产出最终 diff)。与提案权独立——可以让只旁听的仲裁型 agent 做融合,也可以禁止某个提案者自己定稿。
- 最终把「diff 应用进主工作区」的批准权永远在用户,不属于任何 agent(既有铁律)。
- 现有"(可动手)"被动标注在第 ③ 步随写开关退役;引擎无 writeArgs 者两个权限位均不可开(能力上限)。

### 2. 软门清单默认值:无冲突默认接纳

- 与其他提案**无重叠**的 Revision 默认 keep(前提:硬门四条全过);
- **有重叠/冲突**的才列入清单,逐条走第 1 项的讨论-拍板流;
- (推翻仲裁的"全部逐条填":用户判断交互成本大于遗漏风险;硬门与沙箱标注仍兜底。)

### 3. CAS blob 回收:目前全部保留

- 不做 GC,审计完整性优先;磁盘增长问题出现了再说(届时有真实数据可依)。

## 实施顺序(承裁决)

1. **最小第一步**(~30 分钟,零模型调用):deterministicGate 单测(blob 篡改→block、基线漂移→block、无重叠不进软门、同文件重叠→进清单)+ 实现;手造"A 进 B"语义错误 patch,确认结构检查放行而沙箱判败、UI 标注"未覆盖";
2. CAS 存储层(Revision/blob 读写);
3. 处置清单 + 冲突"深入讨论"流(pro/con 生成);
4. 与现有 builds.jsonl 动手流的融合(新机制须为现有动手的超集,不并行双轨)。

## 约束(沿项目既有)

零依赖(sha256 用 node:crypto);Windows 优先;审批权全在用户;写模式调用能省则省;中英双语进字典;落盘/展示过 redactor。
