# 功能候选:仲裁匿名化(Judge Anonymization)

状态:候选(未排期)。来源:2026-07-20 对比 karpathy/llm-council 的评审设计。

## 一句话

仲裁阶段把辩手具名替换为「辩手 A / 辩手 B」,防止仲裁模型因品牌先验(自家模型、名气差异)偏袒某一方——llm-council 用匿名化评审防品牌偏见,这个动机对 Roundtable 的仲裁同样成立。

## 现状与风险

`buildJudge` 现在给仲裁看的是具名材料:`## Claude 的最终陈述`、滚动摘要与分歧分类表里通篇是辩手真名。仲裁模型(常常本身就是参赛的 Claude 或 Codex)存在两类可疑先验:偏袒同门、或按模型名气打分——而仲裁标准要求它只比证据强弱、假设多少、证伪点质量。

## 方案(最小切法)

1. 仅在**仲裁这一步**做匿名:构建 judge 提示词前,把两位辩手名在摘要与最终陈述文本里统一替换为 `Debater A` / `Debater B`(按 roles.debaters 顺序,双语各有对应词);
2. 裁决卡展示时**反向替换回真名**(UI 与落盘都还原),用户视角无感;
3. 辩论各轮保持具名不变——互相点名反驳是交叉质询的机制核心,不能匿名。

## 已知边界与风险(必须在实现时处理)

- **自我指认**:辩手正文里可能自称(「我是 Claude」「as Claude Code…」),简单字符串替换挡不住全部指纹(口吻、格式习惯也会泄露身份)——匿名化降低而非消除偏见,文档与提交信息里不应过度承诺;
- **名字碰撞**:辩手名可能是常见词(如用户自定义 agent 名叫 "Judge"),替换需按词边界并优先长名;
- **反向替换污染**:裁决卡里模型自己写出的 "Debater A" 全部会被还原成真名,属预期行为,但若模型恰好用这个词讨论抽象概念会误伤(概率低,可接受);
- 书记摘要中的「问 Claude:」质询行在仲裁材料里也要一并匿名。

## 与 llm-council 的差异说明(为什么只匿名仲裁)

llm-council 是一次性问答,匿名互评是主流程;Roundtable 的辩论轮次靠具名点名质询驱动,匿名只该发生在「比较与裁决」这个单点上。

## 验证方式

- 单测:含名字的摘要/陈述进 buildJudge 后无真名残留;裁决卡反向替换后无 Debater A/B 残留;
- 效果验证(可选):同一场辩论分别跑具名/匿名仲裁,观察裁决是否翻转——n=1 不具统计力,只作 sanity check。
