# 设计定稿:统一容器——会议成为工作台时间线里的事件(方案 A)

状态:**已定稿,分期实施**。用户 2026-07-21 拍板"直接做 A"。

## 终局心智模型

只有一种顶层容器:**工作台**(场所,持续存在)。会议是工作台讨论流中的一个**升格事件**(有开场、轮次、裁决、终点),以**会议卡**形式存在于时间线;裁决卡自动落回原地。「升格」「回流」作为概念消失——会议本来就发生在讨论里,裁决本来就落在原地。

与 diff 机制同构:工作台时间线是唯一容器,**变更、会议、裁决都是流经它的一等公民对象**(变更卡在右栏账本,会议卡与裁决落在时间线)。

## 为什么(混乱诊断,2026-07-21)

两实体在 UI 中并列呈现,但实为一个工作流的两个阶段:名词过多层级不明(工作台/会议/群聊/互聊)、界面同构语义不同构(场所 vs 事件)、入口并列强迫过早选择、升格/回流是暗道而非动线。

## 分期实施

### 第一期(用户可见的统一)

1. **入口收敛**:侧栏单一「+ 开始」(开工作台);「就此开会」在工作台内部(原升格改名);会议建立页保留(草稿链路与高级直建),从工作台 setup 页脚注入口进入;
2. **侧栏母子分组**:有来源的会议缩进显示在其来源工作台之下;孤儿会议(草稿直建/历史)暂平级;
3. **裁决自动落回**:会议出裁决卡即自动投放到来源工作台时间线(复用 note 注入;重跑裁决则再次落卡,以最新为准);手动「投放到任意工作台」保留(孤儿会议/再分发用);
4. **时间线事件通知**:从工作台开会时,向其时间线注入「会议《X》已开始」记录;
5. 命名:会议内「群聊」改「**场边追问**」,消除与工作台聊天的同名混淆。

### 第二期(结构收编,依赖第一期跑顺)

- 会议卡可点击:时间线中的会议记录变成活卡片(状态、轮次、进入会场);
- 草稿直建的会议自动包一层工作台(消灭孤儿);
- 侧栏只列工作台,会议彻底退出顶层列表;
- 存储迁移:会议目录归入所属工作台目录之下(或保持平级目录+强引用,视 sessions 目录兼容性定)。

### 非目标

- 不改会场三栏沉浸视图本身(它作为"事件的打开态"保留);
- 不动委员会辩论机制。

## 兼容

- 现有平级会议照常打开/恢复;origin 字段已有,母子关系零迁移成本;
- roundtable-meeting 技能的 /api/draft 链路不变。
