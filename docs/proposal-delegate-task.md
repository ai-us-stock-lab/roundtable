# 功能候选:长任务委派(Delegate Task)

状态:候选(未排期)。来源:2026-07-20 英文 demo 录制的真实工作流复盘。

## 一句话

在工作台里指定一个模型,在真实环境执行一条**经用户批准**的长任务(命令),输出流式回到群聊,结束状态成为讨论素材——补齐"说(chat/relay)"与"改文件(动手)"之间缺失的"执行"能力。

## 背景:暴露这个缺口的真实场景

英文 demo 视频共录了三轮,工作流是:Claude(本会话)改代码 → Codex(另一个 CLI 会话)执行录制 → Codex 发现问题 → **用户人肉转述**给 Claude → Claude 修 → 再人肉转述回去。

三轮下来的观察:

- "Codex 报问题 → Claude 修"这段对话,放进工作台群聊可以原文直达、当场追问,转述有真实的信息损耗(例:"31–52 秒路径泄露"只能靠推理定位,拿不到第一手报错原文);
- 但"执行录制"这一步放不进来:工作台的三种能力都不覆盖——
  - **chat/relay**:只说话,不执行;
  - **动手(build)**:隔离副本里改文件、产 diff 审批,不执行长命令;
  - **应用前检查(checkBuild)**:能在副本跑命令,但绑定在某个待批 diff 上、5 分钟硬超时、结果只回报不进讨论流。

## 需求定义

- 用户在工作台输入一条任务命令(或批准模型建议的命令),指定一个执行模型/执行器;
- 在**真实工作区**(默认)或隔离副本(可选)执行,超时可配(默认 30 分钟量级);
- stdout/stderr 流式回聊天(复用 `build-progress` 瞬态事件通道,不进回放缓冲),经现有 redactor 脱敏;
- 结束后一条结构化消息入会话记录(命令、退出码、尾部输出摘要),所有参与者可见、可引用讨论;
- 可随时停止;绝不自动 git commit/push。

## 安全边界(沿用既有契约)

- **命令必须由用户显式输入或逐条批准**——模型可以建议,不能自主执行;
- env 走 adapter 的 `envWhitelist` 同款机制;任务级追加 env(如 `ELEVENLABS_API_KEY`)需要用户在批准时显式勾选,且值不落任何日志(redactor 兜底);
- 输出落盘与 SSE 均过 redact(凭据 + 本机路径,已有);
- 超时/停止后必须走与 demo 管线同款的确定性清理(子进程树终结,句柄不残留)。

## 与现有机制的关系

| 能力 | 环境 | 产物 | 审批点 |
|---|---|---|---|
| chat/relay | — | 观点 | 无需 |
| 动手 build | 隔离副本 | diff | 应用前逐文件审批 |
| 检查 checkBuild | 副本+待批 diff | 通过/失败 | 命令由用户输入 |
| **委派 delegate(本提案)** | 真实工作区(默认) | 执行结果+日志 | 命令逐条批准 |

## 最小可行切法(MVP)

1. 复用 `runCommand` 基建(worktree.js),解除与 build 记录的耦合;
2. 超时改为参数(默认 30 分钟,上限待定),流式输出接 `build-progress` 通道;
3. 工作台 UI 加"委派"入口:命令输入框 + 执行模型选择 + 确认对话框(显示完整命令与生效 env 追加项);
4. 结束消息入 `messages.jsonl`,与普通消息同构。

## 非目标

- 定时/周期任务、任务队列、并行任务池;
- 让模型免批执行任意命令(哪怕沙箱内)。

## 开放问题

- 需要 GUI 的任务(如 demo 录制本身要起 Playwright 浏览器)是否在范围内?无头模式可行,但"录屏检查画面"这类天然要人眼,建议 v1 明确只保证无头命令;
- 巨量输出的截断策略(现 checkBuild 取尾部 8000 字符,长任务可能需要滚动落盘 + 尾部回显);
- 执行中途模型是否可以"看着输出接着说"(流式输出对其他参与者是否构成新的上下文注入点)。
